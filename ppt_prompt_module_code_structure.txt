반도체 CMP 공정 APC 시스템 "MICO 학습 파이프라인 코드 구조" 교육용 PPT를 만들어줘.

[자료의 목적과 청중]
- 청중: 알고리즘(산식) 교육은 이미 받았고, 이제 코드를 직접 읽고 수정해야 하는 사내 엔지니어
- 목적: Common/Module.py의 run()에서 시작해 3개 학습 알고리즘(Pre_Thk_VM → Removal Rate
  → Offset)의 함수 호출 구조를 한눈에 파악하고, "어떤 수정은 어느 파일의 어느 함수만
  보면 되는지" 알 수 있게 하는 코드 워크스루
- 관통 메시지 2개 (자료 전체에서 반복 강조):
  ① "runner → 오케스트레이터 → 계산 클래스" 3층 구조 — 층만 알면 코드를 바로 찾는다
  ② 3개 학습 모듈은 서로 함수를 호출하지 않는다 — 앞 단계가 MongoDB 컬렉션에 저장하고
    뒤 단계의 load_xxx()가 읽는 "컬렉션 사슬"로 연결된다

[디자인 공통 지침]
- 깔끔한 기업 교육자료 스타일, "코드 지도" 느낌
- 함수명·파일명·컬렉션명은 전부 고정폭 폰트
- 호출 트리/시퀀스의 화살표에는 "왜 호출하는지" 짧은 라벨을 달 것
- 데이터(MongoDB 컬렉션)는 원통 도형, 함수/처리는 사각형으로 구분
- 각 알고리즘 파트의 첫 장에는 전체 파이프라인 미니맵을 구석에 두고
  현재 위치를 하이라이트 (지금 어디를 보고 있는지 길을 잃지 않게)
- 장수는 내용에 맞게 자유롭게 구성하되, 아래 순서와 내용을 따를 것

[표지]
- 제목: MICO 학습 파이프라인 코드 구조
- 부제: run()에서 시작하는 함수 호출 지도 — Pre_Thk_VM · Removal Rate · Offset

[코드를 읽는 지도 — 3층 구조와 파일 역할]
- 폴더/파일 지도:
  module/{공정명}/Module.py     ← [1층 runner] 공정 등록. 아래 2줄이 전부:
                                   from Common.Module import run
                                   run('DRAM', 'M1 CU CMP')
  Common/Module.py              ← [2층 오케스트레이터] run() + Module_Get 클래스
                                   키 분해, 그룹 분기, 파이프라인 순서, 저장, 에러 알림
  Common/PRE_THK_VM.py          ← [3층 계산] PRE_THK_VM_Get   (사전 두께 VM)
  Common/REMOVAL_RATE.py        ← [3층 계산] Removal_Rate_Get (연마율 회귀)
  Common/OFFSET.py              ← [3층 계산] OFFSET_Get       (상태별 시간 보정)
  Common/Get_Data.py            ← [공통 조회] Set-up·MongoDB·파라미터 매핑 전담
- 층별 규칙 표:
  | 층 | 아는 것 | 모르는 것 | 이 층을 수정하는 경우 |
  | runner | 공정 이름뿐 | 로직 전부 | 신규 공정 추가 |
  | 오케스트레이터 | 처리 순서·저장처·Set-up 구조 | 산식 내용 | 처리 흐름·경로 조건 변경 |
  | 계산 클래스 | DataFrame in/out 계산 | DB·Set-up 존재 | 산식 변경 |
- 포인트: 모든 설정(pol_type, 그룹, 사전공정, Target, 알람 채널)은 코드가 아니라
  웹 Set-up DB에 있음 → runner에 config조차 없는 이유

[run() 진입 호출 트리]
- 호출 트리 (위→아래, 각 노드에 목적 한 줄):
  run(family, oper_desc)                      ← 최상위 진입점
  ① Get_data.baseinfoGetData()                : 웹 Set-up 조회 → mico_info_table
  ② for_key_list 생성                          : Lot_Code + Oper_Code + Fab 조합
                                                → 학습의 처리 단위(키) 정의
  ③ Group_Name 값으로 분기 (Set-up에서 옴):
    ├ 그룹 미지정 → _run_single(table, keys)
    │   키별 루프:
    │   · Module_Get.fetch_merge_data()       : Merge 데이터 조회 (0행이면 skip)
    │   · operation_id 필터 (0행이면 skip)     ← 파이프라인 진입 전 2중 방어
    │   · _run_pipeline(merge_df, key)
    └ 그룹 지정   → _run_grouped(table, group_name)
        · 그룹 내 모든 키의 fetch_merge_data() 결과를 먼저 concat
        · 키별로 같은 합산 merge_df로 _run_pipeline(use_group_rr=True)
        : 데이터가 적은 공정을 여러 Lot_Code로 묶어 학습하기 위함
- pol_type은 _run_pipeline이 Set-up(Pol_Type 컬럼)에서 읽어 인자로 내려줌
  → 하위 함수는 전역/DB 접근 없이 인자만으로 동작
- 에러 처리 구조 (3겹 try/except):
  run() 전체 / _run_pipeline() 단위 / 각 compute_* 내부
  → 실패 시 Cube 메신저로 traceback 발송 후 다음 키·다음 단계 계속
  (한 공정 실패가 전체 배치를 멈추지 않게 하는 격리 설계)

[전체 파이프라인 모식도 — 이 자료의 핵심 한 장, 가장 크게]
- _run_pipeline()의 실행 순서 4단계와 컬렉션 사슬을 하나의 그림으로:
  [Merge Hub가 미리 생성해 둔 입력 - 원통]
    MICO_Merge_df_...        (CMP 실측 + 사전공정 병합)
    MICO_PRE_THK_INFO_...    (사전공정 두께)
    EQPM (장비 이벤트)
         ↓ fetch_merge_data()
  1. Module_Get.compute_pre_thk_vm()   ──저장→ (원통) MICO_PRE_THK_..._Period
         ↓ Removal_Rate_Get.load_pre_thk_data()가 읽음    ← "함수 호출이 아니라 컬렉션으로 연결"
  2. Module_Get.compute_removal_rate() ──저장→ (원통) MICO_Removal_Rate_...
     (그룹이면 compute_removal_rate_group / EQPM은 _build_eqpm_df()로 조회)
         ↓ OFFSET_Get.load_rr_data()가 읽음
  3. Module_Get.compute_offset()       ──저장→ (원통) MICO_OFFSET_...
     (그룹이면 compute_offset_group)
         ↓
  4. Module_Get.check_alarm()          : 세 컬렉션을 모두 읽어 N-Sigma 점검 → 메신저 알람
  [최종 소비처] APC 연마 시간 계산 / Simulation / 웹 모니터링
- 왜 이 순서인가: 2가 1의 결과를, 3이 2의 결과를 입력으로 쓰기 때문
  → 순서를 바꾸면 이전 실행의 낡은 학습값을 읽게 됨
- 컬렉션명 규칙: MICO_{종류}_{Lot_Code}_{Oper_Desc}_{Fab}
  → _make_pre_thk_mongo() / _make_rr_mongo() 등 헬퍼가 규칙을 한 곳에 고정
- 느슨한 결합의 이점: 단계별 재실행·단독 테스트 가능

[내부 구조 — 플래그 3개가 호출 경로를 결정한다]
- 함수 위치: Common/Module.py의 Module_Get 클래스
  시그니처: compute_pre_thk_vm(merge_df, mico_info_key, pol_type)
- 도입부: _make_pre_thk_mongo()로 저장 핸들 생성,

- Thk_Para별 루프 (같은 키 안에 두께 파라미터가 여러 개일 수 있어 루프 필요):

  (셋 다 False → continue로 스킵)
- 경로별 호출 시퀀스 (3열 비교 모식도로):
  [ITM 경로]                [Detrend 경로]                    [회귀 전용 경로]
  BIAS 계산 (인라인)         key_df 행별 루프:                  BIAS 계산 (인라인)
  ↓                         · Get_data.PadParaGet()           ↓
  iqr_filter()              : 패드 사용량 컬럼명 조회            (집계 없음,
  ↓                         · Get_data.APCParaGet()            회귀용 y만 준비)
  rolling_mean()            : 연마시간 컬럼 목록 조회

    detrend 입력(연마시간 컬럼, 목표값)이 행 단위로 결정되기 때문
  · 왜 iqr_filter가 rolling_mean보다 먼저인가: 이상치가 이동평균에 섞이기 전에 제거
- 공통 마무리 (경로 무관):
  ① has_regression이면: MICO_PRE_THK_INFO 컬렉션 직접 로드

  ② mongo.push_df(pre_thk_table)        : 결과 저장

[PRE_THK_VM_Get 계산 함수 4개 명세]
- 핵심: PRE_THK_VM.py는 "DataFrame in → DataFrame out" 순수 계산 함수만 모음.
  DB·Set-up을 모르므로 단독 테스트 가능, Module.py가 조립 담당.
- 명세 표:
  | 함수 | 입력 → 출력 | 호출 위치 | 존재 이유 |
  | compute_detrend(merge_df, APC_Para_merge, Thk_Para, Pre_Target, Post_Target, Pad_Para, use_pressure) | 실측 → Detrend_Thk 컬럼 추가 | Detrend 경로의 행별 루프 안 | RR 역산→패드 사이클 분리→추세 제거. 가장 복잡해서 별도 함수로 격리 |
  | iqr_filter(df, col, sigma=3) | df → 이상치 행 제거 | ITM·Detrend 경로 (rolling_mean 직전) | 이상치 제거 방식을 한 곳에 고정. col 인자로 BIAS/Detrend_Thk 공용 |
  | rolling_mean(df, value_col, Pre_Thk_Period, min_count) | df → Pre_Thk/Pre_Thk_Count 추가 | ITM·Detrend 경로 | 전공정 장비·채널(pre_eq_ch)별 기간 이동평균. value_col 인자로 두 경로 공유 |
  | fit_pre_oper_regression(..., oper_pairs, y_col) | 학습 df + 사전공정 df → b1/b0 컬럼 추가 | 모든 경로 마지막 (has_regression) | 사전공정 2~4를 리스트 하나의 루프로. y_col 인자로 경로별 y(BIAS/Detrend_Thk) 흡수 |
- 설계 포인트: 두 경로가 후처리 함수를 공유하는 비결 = 컬럼명을 인자(value_col, y_col)로
  넘겨 분기를 흡수 (if문 중복 제거)
- 수정 가이드: 산식 변경 → PRE_THK_VM.py만 / 경로 조건·저장 변경 → Module.py만

PART 2. compute_removal_rate() — 파이프라인 2단계
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[진입과 데이터 준비 — compute_rr()를 부르기까지]
- 호출 트리:
  compute_removal_rate(merge_df, mico_info_key, pol_type)   ← Module.py
  ① VM 준비 — Set-up 값으로 3분기 (1단계 결과를 소비하는 지점):
     | 조건 | 호출 함수 | 하는 일 |
     | Pre_Oper_Code도 Pre_Oper_Code2도 없음 | (인라인) | 모든 Thk_Para에 _VM=0 컬럼 생성 |
     | Pre_Oper_Code 없음 + Code2 있음 | apply_pre_oper2_correction() | VM=0에서 시작해 회귀 보정만 가산 |
     | Pre_Oper_Code 있음 | load_pre_thk_data() | Period 컬렉션 학습값을 merge_asof 결합 + 회귀 보정 |
  ② _build_eqpm_df(merge_df_rr, Maker, Fab)                 ← Module.py 헬퍼
     : Get_data.EQPMGetData_HUB()로 장비 이벤트 조회 후
       compute_rank() 내부 함수로 PM 후 누적 웨이퍼 수(rank) 계산
     : Maker별(AMAT/EBARA/KCT) 채널 컬럼 구성도 여기서
  ③ _make_rr_mongo()                                        : 저장 핸들
  ④ search_key(Set-up 행)별 루프:
     BIAS 계산(인라인) → Removal_Rate_Get.compute_rr(merge_df_rr, key,
                          pol_type, EQPM_df, RR_Alarm, mongo)
- single vs group: 그룹은 Lot_Code별로 load_pre_thk_data()를 각각 호출해 concat
  (컬렉션이 Lot_Code별이므로), 마지막 호출이 compute_rr_group()
- 포인트: EQPM_df와 mongo를 위에서 만들어 인자로 내려줌
  → compute_rr는 조회·저장처를 몰라도 됨 (의존성 주입 패턴)

[compute_rr() → _process_models() — 루프 구조와 모델 함수 4개 (PART 2의 핵심 모식도)]
- compute_rr()의 역할은 "준비만":
  ① Get_data.APCParaGet/HeadParaGet/PadParaGet/DiskParaGet()
     : APC_Para 이름 → 실제 컬럼명 목록 변환 (매핑 지식을 Get_Data에 격리)
  ② Set-up RR_Para 값(HEAD/PAD/DISK)으로 consumable_Para(x축) 선택
  ③ 필터: operation_id + recipe_id 일치, IDLE 제외
  ④ _detect_cycles()   : 장비//레시피별 소모품 급감(diff < −3)으로 cycle 부여 (최신=1)
  ⑤ _process_models(..., group_mode=False)   ← 실제 계산은 전부 여기
- compute_rr_group()도 동일 골격, 차이는 recipe 필터 없음 + group_mode=True
  → single/group이 _process_models() 하나를 공유, group_mode 인자로 분기 (중복 제거)
- _process_models() 내부 = 2중 루프 + 모델 함수 4개 (모식도로):
  for eqp_model in ...:                     ← 장비 모델별 (4분위 경계 계산)
    RR 컬럼 계산(인라인), 6σ 이상치 제거
    for eq_recipe (그룹: rr_key) in ...:    ← 장비//레시피 (그룹: 장비//그룹명) 별
      10건 미만 continue
      _check_rr_alarm()                     : TIME 파라미터면 최근 3건 ±3σ 감시
      quartile 균형 조건 통과 시:
        _fit_lr()                           : ① 기본 모델 (b1, b0)
        _fit_weighted()                     : ② 가중 모델 ('-' 반환 가능)
        _get_current_cycle() → _fit_current(): ③ 현재 사이클 모델
          └ 내부에서 _get_pm_rank()          : PM 후 웨이퍼 수 확인 (성립 조건)
          └ 내부에서 _fit_lr() 재사용
        _fit_if()                           : ④ IF 모델 (Pad_Seperation 구간)
          └ 내부에서 _fit_lr() 재사용
        report dict → '-' 값 필드 제거 → mongo.insert_row(report)
- 설계 포인트: 모델 함수들이 조건 미달을 예외가 아닌 '-' 반환으로 처리
  → 호출부는 분기 없이 받아 저장 직전에 '-'만 걸러냄 (부분 성립 허용 구조)

PART 3. compute_offset() — 파이프라인 3단계
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[진입 — 얇은 오케스트레이션]
- 호출 트리:
  compute_offset(merge_df, mico_info_key, pol_type)          ← Module.py
  ① mico_info_key를 FB_Type=='TIME'으로 필터                  ← 첫 줄에서 대상 축소
     : Offset은 연마 "시간" 보정이므로 TIME 파라미터만 대상
     (하위 함수는 이 규칙을 몰라도 됨)
  ② OFFSET_Get.load_rr_data(merge_df, ..., APC_Para_List, ...)
     : 2단계 결과를 소비하는 연결 지점.
       merge_df에 {APC_Para}_B1/_B0 컬럼이 붙어서 돌아옴
  ③ search_key 행별로 OFFSET_Get.compute_offset() 호출
     : 리스트 컴프리헨션으로 수집 → None(데이터 없음) 걸러냄 → 전부 None이면 조기 return
  ④ 유효 결과 concat → OFFSET_Get.compute_lc_offset(temp_df, ...)
- single vs group: 그룹은 행별 루프 대신 compute_offset_group() 1회 (레시피 무관 합산),
  이후 compute_lc_offset()은 동일
- 포인트: Module.py 쪽이 앞 두 단계보다 훨씬 얇음
  → 준비할 외부 데이터(EQPM 같은)가 없고 계산·저장이 전부 OFFSET_Get 안에 있어서

[OFFSET_Get 내부 — 계산 1번에 저장 2번, 반환값이 이어달리기 (PART 3의 핵심 모식도)]
- 구조 그림:
  compute_offset()           compute_lc_offset()
  ┌────────────────┐         ┌─────────────────┐
  │ OFFSET 계산     │ temp_  │ LC/TB 분리        │
  │ Idle/Layer 집계 │─data──→│ 라벨 정규화        │
  │ → 1차 저장      │ 반환    │ _build_idle_table│
  └────────────────┘         │ → 2차 저장       │
                             └─────────────────┘
- compute_offset() 내부 시퀀스:
  ① Get_data.APCParaGet() / PadParaGet()    : 컬럼명 매핑 (앞 단계와 동일 패턴)
  ② operation_id + recipe_id 필터 → 비면 return None  ← 호출부 None 걸러내기와 짝
  ③ B1/B0 결측 보충: 같은 eq_recipe 평균 → 전체 평균 순 (인라인)
  ④ RR / RR_Pad / OFFSET 계산 (인라인) + clip(−5, 3) + LC_T_/LC_TB_는 0
  ⑤ Idle_/Layer_ 라벨만 필터 → (장비,레시피,IDLE) 5건 이상 그룹만 평균
  ⑥ mongo.push_df(idle_table)               ← 1차 저장 (Idle/Layer OFFSET)
  ⑦ return temp_data                        ← LC 계열 행이 남은 원본 반환 = 2차 재료
- compute_offset_group() 차이 2가지: ⑤가 레시피 무관 (장비,IDLE) 집계 /
  저장 직전 그룹 내 모든 레시피에 동일 OFFSET 복제
- compute_lc_offset() 내부 시퀀스:
  ① IDLE 라벨 패턴으로 LC 그룹 / TB 그룹 분리 (contains 조합)
  ② Offset_Group=='Y'면 라벨에서 장비 번호 토큰 제거 (여러 장비 데이터 통합용)
  ③ _build_idle_table(combined)             : 전체 조합 생성 + 2단 평균
  ④ mongo.push_df(lc_offset)                ← 2차 저장 (LC/TB OFFSET)
- 왜 2단 구조인가: Idle/Layer는 레시피 단위 산출물이라 루프 안에서 바로 저장,
  LC는 전체 레시피 결과를 합쳐야 계산 가능해서 반환값을 모아 한 번에 처리

[OFFSET_Get 함수 명세]
  | 함수 | 입력 → 출력 | 호출 위치 | 존재 이유 |
  | load_rr_data(...) | 실측 df → B1/B0 컬럼 붙은 df | Module 초입 | RR 결과 소비 지점. pivot + merge_asof(시점별 계수 매칭) 격리 |
  | _get_b_coef(row) | RR 행 → (b1_new, b0_new) | load_rr_data 안 (apply) | "weighted 우선, 없으면 기본" 계수 선택 규칙을 한 곳에 |
  | compute_offset(...) | df + Set-up 행 → 1차 저장 + temp_data 반환 | Module 행별 루프 | 레시피 단위 OFFSET 계산·Idle/Layer 집계 본체 |
  | compute_offset_group(...) | df + Set-up 전체 → 동일 | Module (그룹, 1회) | 레시피 무관 집계 + 복제 저장 변형 |

  LC 라벨 분류 → compute_lc_offset / 집계 방식 → _build_idle_table
